## 3. 用户体验审计

### 3.1 用户旅程分析

**核心旅程：上传文档 → 知识编译 → Wiki 浏览 → Q&A 查询**

| 步骤 | 后端能力 | 前端支持 | 体验断点 |

| 3. 浏览 Wiki 页面 | ✅ GET /llm-wiki/page | ✅ F5 Wiki 浏览 | 🟡 Markdown 渲染简陋 |
| 4. 双向链接导航 | ✅ GET /backlinks | 🟡 手动点击 | **`[[wikilink]]` 未渲染为链接** |
| 5. Wiki Q&A | ✅ POST /llm-wiki/query | ❌ 无 Q&A 界面 | **核心功能不可达** |
| 6. 健康检查 | ✅ POST /llm-wiki/lint | ❌ 无 Lint 界面 | **用户无法看到知识库健康状态** |
| 7. 漂移监控 | ✅ GET /llm-wiki/stale | ❌ 无监控界面 | **用户无法感知知识过时** |

### OpenTelemetry 分布式追踪交付摘要（Sprint 10+ P3-3）

**目标**：在已交付的 Prometheus 指标基础上，补齐 P3-3 可观测性的另一半 — OpenTelemetry 分布式追踪，支持 W3C Trace Context 自动传播、自定义业务 span、日志关联与 OTLP 导出。

**交付内容**：

1. **追踪模块** [observability/tracing.py](file:///workspace/backend/app/observability/tracing.py)
   - **`setup_tracing(app, provider=None)`**：初始化 OpenTelemetry（幂等），自动 instrumentation FastAPI（入站 server span）+ httpx（出站 client span）
   - **`span(name, **attrs)` 上下文管理器**：业务代码零侵入包裹关键路径，未启用时为 no-op（yield None），业务无需条件判断
   - **`get_current_trace_id()` / `get_current_span_id()`**：获取当前活跃 span 上下文（16 进制 32/16 字符），用于日志关联与错误响应注入
   - **`tracing_log_processor`**：structlog processor，将当前 span 的 trace_id/span_id 自动注入日志事件，实现"日志-追踪"双向关联
   - **OTLP 导出**：`OPSKG_OTLP_ENDPOINT` 配置时通过 BatchSpanProcessor + OTLPSpanExporter 发送到 Jaeger / Tempo / OTel Collector；未配置时降级为 ConsoleSpanExporter
   - **Resource 标识**：service.name / service.version / deployment.environment 自动注入
   - **`_reset_for_test()`**：测试专用重置方法

2. **业务 span 埋点** [core/llm/openai_compat.py](file:///workspace/backend/app/core/llm/openai_compat.py)
   - LLM `chat()` 方法用 `span("llm.chat", backend, model, message_count)` 包裹
   - 惰性导入避免循环依赖，未启用时 no-op
   - 后续可在文档解析、知识编译、webhook 投递等关键路径扩展

3. **/tracing/status 端点** [main.py](file:///workspace/backend/app/main.py)
   - 返回 `{enabled, trace_id, span_id}`，便于调试与运行时检查
   - 启用时 trace_id/span_id 反映当前请求的 span 上下文

4. **配置** [requirements.txt](file:///workspace/requirements.txt)
   - 新增 5 个 opentelemetry 依赖（api / sdk / instrumentation-fastapi / instrumentation-httpx / exporter-otlp）
   - 默认关闭，`OPSKG_TRACING_ENABLED=1` 启用

5. **__init__.py 导出** [observability/__init__.py](file:///workspace/backend/app/observability/__init__.py)
   - 新增 6 个公开 API：`setup_tracing` / `span` / `get_tracer` / `get_current_trace_id` / `get_current_span_id` / `tracing_log_processor`

6. **验证脚本** [verify_tracing.py](file:///workspace/scripts/verify_tracing.py)
   - 10 个测试组、**39 个验证点全部通过**
   - 覆盖：未启用 no-op / setup_tracing 注入 InMemorySpanExporter / 幂等 / span() 生成+属性 / trace_id/span_id 获取 / tracing_log_processor 日志关联 / FastAPI 自动 instrumentation（TestClient 端到端）/ 业务 span（llm.chat）/ 嵌套 span 父子关系 / 容错降级 / 未启用不初始化

**关键设计决策**：
- **默认关闭**：`OPSKG_TRACING_ENABLED=0` 默认值，零开销；生产环境按需开启
- **no-op 降级**：未启用时 `span()` yield None，`get_current_trace_id()` 返回 None，业务代码无需任何条件判断
- **容错守护**：所有 otel 调用 try-except，tracing 异常绝不影响业务（与 metrics 设计原则一致）
- **OTLP 优先 + Console 兜底**：配置 endpoint 走 OTLP（生产），未配置走 Console（开发调试）
- **自动 instrumentation**：FastAPI + httpx 自动生成 server/client span，业务代码只需包裹关键路径
- **日志关联**：structlog processor 自动注入 trace_id，Grafana / Loki 可通过 trace_id 跳转到 Jaeger / Tempo
- **惰性导入**：LLM 模块用 `_tracing_span()` 工厂函数惰性导入，避免 `core → observability → core` 循环依赖
- **测试可注入**：`setup_tracing(app, provider=...)` 支持注入 InMemorySpanExporter，验证脚本可端到端断言 span

   - **重试**：指数退避 [30s, 120s, 600s]，默认 1+3=4 次尝试
   - **后台 worker**：每 15s 扫描到期重试任务（FastAPI lifespan 启停）

   - `POST /webhooks/{sub_id}/rotate-secret` 重置签名密钥
   - `POST /webhooks/{sub_id}/test` 发送 `webhook.test` 测试事件
   - `GET /webhooks/{sub_id}/deliveries` 订阅级投递记录
   - `GET /webhooks/deliveries` 全局投递记录（可按 status 过滤）

   - 9 个测试组、**62 个验证点全部通过**
   - 含端到端 mock HTTP server 测试（成功投递 + 签名验证 + 500 重试场景）
   - 含后台 worker 启停测试
   - 含同步上下文 dispatch 不阻塞测试

**关键设计决策**：
- **fire-and-forget**：`dispatch_event` 用 `asyncio.create_task` 后台投递，业务请求绝不阻塞
- **容错**：webhook 子系统任何异常都被捕获并 log，绝不影响业务流程
- **secret 安全**：仅在创建/重置时返回一次明文，list/get 接口永不返回
- **可观察**：每次投递都持久化状态（pending/retry/success/failed/skipped）+ 响应码 + 响应体（截断 4KB）
- **HTTP headers**：`X-Webhook-Signature` / `X-Webhook-Event` / `X-Webhook-Event-Id` / `X-Webhook-Timestamp`，便于接收方验签与去重

### HA 高可用交付摘要（Sprint 10+ P3-4）

**目标**：为 OpsKG 后端提供 K8s 友好的健康探针 + 实例标识，支持多副本部署下的流量调度与故障自愈。

**交付内容**：

1. **HA 模块** [ha/__init__.py](file:///workspace/backend/app/ha/__init__.py)
   - `get_instance_id()` — 优先级：`OPSKG_INSTANCE_ID` env > `settings.instance_id` > `hostname+pid` 自动生成
   - `get_startup_time()` — 进程启动时间（ISO8601 UTC），用于计算 uptime
   - `check_db_connection(db_path)` — SQLite 只读模式（`mode=ro`，timeout=2s）连接检查，返回 `{ok, path, error, table_count}`
   - `collect_health()` — 收集 6 个 DB（documents/versions/webhooks/events/search/templates）状态 + webhook worker 状态 + uptime，返回 `status: ok|degraded|down`
   - `collect_readiness()` — 严格检查：所有 DB ok + webhook worker running，返回 `ready: bool` + `checks` 明细

2. **配置扩展** [config.py](file:///workspace/backend/app/config.py)
   - `instance_id: str = ""` — 实例 ID（空则自动生成）
   - `deployment_mode: str = "standalone"` — `standalone` | `replicated`（多实例 + 共享存储）

   - **lifespan 启动日志**：记录 `instance_id` / `deployment_mode` / `started_at`，便于多实例日志关联
   - **`/health` 增强**：返回 `collect_health()` 完整结构（status / instance_id / deployment_mode / uptime_seconds / started_at / dbs / background_workers），HTTP 200 表示进程存活
   - **`/ready` 新增**：返回 `collect_readiness()`，`ready=False` 时返回 HTTP 503，K8s 自动从 Service endpoints 摘除流量

4. **验证脚本** [verify_ha.py](file:///workspace/scripts/verify_ha.py)
   - 9 个测试组、**32 个验证点全部通过**
   - 含 instance_id 三级优先级验证、DB 不存在/存在双场景、/ready 503 失败场景、lifespan 日志验证

**关键设计决策**：
- **liveness vs readiness 分离**：`/health` 宽松（任一 DB ok 即 degraded 而非 down，进程存活就返回 200）；`/ready` 严格（所有 DB ok + worker running 才 ready），符合 K8s 探针语义
- **向后兼容**：单实例默认行为不变，多实例需 `OPSKG_DEPLOYMENT_MODE=replicated` + 共享存储（NFS 或类似）
- **DB 检查容错**：只读模式 + 2s 超时，单 DB 不可达不影响其他 DB 检查，所有异常被捕获
- **实例标识三级降级**：env > settings > hostname+pid，确保任何环境下都有非空 instance_id

**K8s 部署示例**：
```yaml

  periodSeconds: 30

  periodSeconds: 10
```

### OIDC SSO 完整接入交付摘要（Sprint 10+ P3-1）

**目标**：在已交付的认证框架（用户名密码 + RBAC + session）基础上，集成标准 OIDC / OAuth2 协议，支持 Google / GitHub / Keycloak 等外部身份提供者，实现单点登录。

**交付内容**：

1. **OIDC 客户端模块** [auth/oidc.py](file:///workspace/backend/app/auth/oidc.py)
   - **`OIDCProvider` 类**：封装单个 IdP 配置（name / display_name / client_id / client_secret / discovery_url / scopes），运行时缓存 discovery 结果（authorization_endpoint / token_endpoint / userinfo_endpoint）
   - **`parse_providers(json_str)`**：从环境变量 `OIDC_PROVIDERS`（JSON 数组）解析多 IdP 配置，支持 Google / GitHub / Keycloak 等任意 OIDC 兼容 IdP
   - **PKCE 支持**：`generate_pkce()` 生成 S256 code_verifier + code_challenge，防止授权码拦截攻击
   - **State 一次性存储**：`save_state` / `pop_state` 内存存储（TTL 10 分钟），CSRF 防护 + 一次性消费
   - **OIDC 用户映射**：`oidc_mappings` 表（provider + oidc_sub → user_id），首次登录自动创建本地用户（角色由 `oidc_default_role` 配置），二次登录按 sub 查找
   - **Token 交换**：`exchange_code()` 用 authorization code 换取 access_token + id_token，优先从 userinfo 端点获取用户信息，fallback 解析 id_token claims
   - **JWT claims 解析**：`_decode_id_token_claims()` base64 解码 JWT payload（不验证签名，仅作 userinfo 不可用时的 fallback）

2. **OIDC 路由** [routers/oidc_router.py](file:///workspace/backend/app/routers/oidc_router.py)
   - `GET /auth/oidc/status` — 检查 OIDC 是否启用 + 提供者列表（不含 secret）
   - `GET /auth/oidc/providers` — 列出已配置的 OIDC 提供者
   - `GET /auth/oidc/{provider}?redirect=/dashboard` — 重定向到 IdP 授权页（带 PKCE + state + redirect_uri）
   - `GET /auth/oidc/{provider}/callback?code=...&state=...` — IdP 回调：
     1. 验证 state（一次性消费 + provider 匹配）
     2. 用 code + PKCE verifier 换取 access_token
     3. 获取用户信息（userinfo 或 id_token claims）
     4. `find_or_create_user` 查找/创建本地用户
     5. 签发本地 session token
     6. 302 重定向到前端 `/login/callback?token=...&redirect=...`
   - **错误处理**：所有失败场景（state 无效 / token 交换失败 / 用户禁用等）均重定向到前端并附带 `?error=...&error_description=...`

3. **配置扩展** [config.py](file:///workspace/backend/app/config.py)

   - `oidc_default_role: str = "viewer"` — OIDC 用户首次登录的默认角色
   - `oidc_redirect_base_url: str` — 后端公网回调基址（空则从请求推断）

- **PKCE 强制**：所有 OIDC 流程都带 PKCE S256 challenge，即使 client_secret 已配置也启用（defense in depth）
- **State 一次性消费**：防止 CSRF + 重放攻击，10 分钟 TTL
- **用户映射表**：`oidc_mappings`（provider, oidc_sub）→ user_id，支持同一 IdP 多账号 + 同一账号多 IdP

### 3.4 产品经理结论

**核心判断**：产品广度已具备竞争力，但 **LLM Wiki 的"知识复利"核心卖点未闭环**。继续扩展 AIOps/SSO 广度，不如先补齐 Wiki 范式深度，让核心差异化真正成立。

**调整建议**：
1. **Sprint 12 优先补齐 Query 回写 + Ingest 反向回链**，让"知识复利"从口号变现实
2. **Sprint 13 补 SAML/LDAP**，打开企业市场（OIDC 已覆盖互联网公司，SAML/LDAP 覆盖传统企业）
3. **前端权限守卫 + 全局 loading bar**，提升 UX 完成度
4. **矛盾检测语义化升级**，当前正则比对覆盖面有限，可引入 LLM 辅助矛盾判定

## 四、Ingest 反向回链实现

## 五、矛盾检测语义化实现

**S12-3** 在原有 regex 快速检测（数值型冲突高精度）之上，新增 LLM 辅助的语义矛盾检测层：

```
候选对筛选（共享 tag 或 outlink）→ 限制 max_pairs 控制 LLM 成本
  → LLM 判断每对页面是否存在事实矛盾 → 返回 JSON [{summary, evidence_a, evidence_b}]
```

关键设计：
- 候选对筛选避免 O(N²) LLM 调用
