## 一、执行摘要

OpsKG 是一个基于 **Karpathy LLM Wiki 范式**的运维知识管理平台，定位为"OKF 一等公民 + 运维增强平台"。本次审计从 AI 工程师与 Agent Harness 专家视角，对项目进行了全量代码审查。

| **知识编译范式** | 生产级 | ★★★★☆ | Karpathy LLM Wiki 三要素（编译/复利/漂移）完整落地 |

### 综合定位

OpsKG 是一个 **"高质量的 RAG + 知识编译系统"**，而非 **"自主 Agent 系统"**。它的核心范式是：
- LLM 作为**编译器**（raw → wiki）和**回答者**（wiki → answer），不是自主决策者
- 工具调用由**外部 MCP 客户端**驱动，LLM 不在 agent loop 中自主选工具
- AIOps 是**独立算法引擎**，LLM 是分析辅助

在 **RAG/知识管理维度**，它已达到生产级；在 **Agent 自主性维度**，还缺少 agent loop、function calling 原生支持、LLM 调用弹性容错（重试/降级/限流）。

## 三、知识编译系统审计（核心）

### 3.1 Karpathy LLM Wiki 范式落地评估

这是项目的核心价值。审计确认三要素均已落地：

#### 3.1.1 编译质量 — 基本落地

- **LLM 参与编译**：[wiki_compiler.py:366-419](file:///workspace/backend/app/knowledge/wiki_compiler.py) 实现 system+user prompt，temperature=0.2（低创造性），要求严格遵循 AGENTS.md 骨架
- **置信度门控**：[wiki_compiler.py:349-353](file:///workspace/backend/app/knowledge/wiki_compiler.py)，`confidence < 0.60` → `review_status: review_needed`
- **LLM 不可用兜底**：[wiki_compiler.py:435-476](file:///workspace/backend/app/knowledge/wiki_compiler.py) 生成模板化正文，由 [wiki_lint.py:141](file:///workspace/backend/app/knowledge/wiki_lint.py) 的 `TEMPLATE_PLACEHOLDER_RE` 检测
- **增量合并**：[wiki_compiler.py:550-590](file:///workspace/backend/app/knowledge/wiki_compiler.py) 保留已有 frontmatter，仅追加 source（去重 by doc_id）
- **反向回链**：[wiki_compiler.py:729-795](file:///workspace/backend/app/knowledge/wiki_compiler.py) 新建页面时扫描已有页面插入 `[[slug|原文]]`

**局限**：
- 增量合并仅追加不整合（`_merge_existing` 行 550-590），长期可能导致"增量补充"章节堆积
- Stale 检测基于属性键字符串匹配（行 576-582），粗糙易误判
- LLM 幻觉无防御机制，仅靠 prompt 约束"不要编造"

#### 3.1.2 知识复利 — 已落地

- [wiki_query.py:866-1049](file:///workspace/backend/app/knowledge/wiki_query.py) `writeback_new_facts`：从 Q&A 回答中 LLM 提取新事实 → 回写到对应 wiki 页面 `## 知识复利补充` 章节 → 标记 `review_status: review_needed`
- 流式与非流式 Q&A 均支持回写
- 回写记录返回给调用方（`writebacks` 字段）

**风险**：回写事实未经校验即持久化，LLM 幻觉可能污染知识库；仅标记 review_needed 但无主动通知机制。

#### 3.1.3 漂移修正 — 完整闭环

完整闭环已落地，对齐 AGENTS.md §八 8.1：

```
检测（detect_drift, checksum 对比）
  → 标记（mark_pages_stale, frontmatter stale=true）
  → 发现（list_stale_pages + Lint TYPE_STALE）
  → 修正（auto_recompile_stale, force 重编译 + diff）
  → 审查（推 ReviewQueue 人工裁定）
  → 清除（clear_stale）
```

入口：`POST /llm-wiki/recompile-stale`

### 3.2 Wiki 健康检查

[wiki_lint.py](file:///workspace/backend/app/knowledge/wiki_lint.py) 实现四类核心检测 + 三类扩展检测：

| 检测类型 | 实现 | 严重度 |
|---------|------|--------|
| 矛盾检测 | regex 抽取"参数: 值"按参数名分组比较 + LLM 语义矛盾检测（可选，max 10 对） | ERROR（数值冲突）/ WARN |
| Stale 检测 | 复用 wiki_drift.list_stale_pages() | WARN |
| Orphan 检测 | 复用 wikilink.get_orphan_slugs() | INFO |
| Missing Concept | 复用 wikilink.get_all_deadlinks()，聚合被引次数 | WARN |
| 缺失章节 | 按 REQUIRED_SECTIONS 校验类型必含章节 | WARN/INFO |
| 空章节 | 检测模板兜底标记 | INFO |
| OKF 违规 | 集成 okf_validator.validate_wiki() | ERROR/WARN |

**忽略机制**：[wiki_lint.py:73-81](file:///workspace/backend/app/knowledge/wiki_lint.py) `compute_issue_key = sha1(type|slug|message)[:16]`，幂等且跨调用稳定，持久化在 SQLite `lint_ignores` 表。

### 3.3 OKF 互操作

[okf_adapter.py](file:///workspace/backend/app/knowledge/okf_adapter.py) + [okf_validator.py](file:///workspace/backend/app/knowledge/okf_validator.py) 实现 OKF v0.1 对齐：

- **三硬性约束对齐**：编译期即生成合规 frontmatter（含 `type` 字段），`wiki_index.py` 生成 `type: index`，`wiki_log.py` 生成 `type: log`
- **双轨链接转换**：内部 `[[wikilink]]`（性能优）↔ 导出 `[display](/dir/slug.md)`（互操作优）
- **导出/导入 bundle**：tarball 导出 + permissive consumption 导入（容忍未知 type、缺失字段、断链）
- **校验三入口**：`validate_concept`（单文件）/ `validate_wiki`（内部 DB）/ `validate_bundle`（目录树）

### 3.4 关键架构问题

#### 问题 1：Wiki 系统与 Neo4j 图谱完全割裂（中）

**现象**：AGENTS.md §九 描述 GraphStore 为"wiki 编译的输入之一"，但实际代码中：
- `wiki_compiler.py` 不导入 `graph_store`
- `wiki_query.py` 不导入 `graph_store`
- GraphStore 仅被 `compiler.py`（旧 KnowledgeCompiler）和 `review_queue.py` 使用

**影响**：系统实质是纯 Markdown Wiki 引擎，而非"图谱增强的 Wiki"。`ops_ontology.yaml` 定义的 12 类实体 + 11 类关系未被 Wiki 系统消费。

**两套编译器并存**：`compiler.py`（面向 Neo4j，去重+合并+权威评分）与 `wiki_compiler.py`（面向 Markdown Wiki）无统一编排。

#### 问题 2：知识复利回写无校验（中）

LLM 提取的"新事实"直接持久化到 wiki 页面，仅标记 `review_needed` 但不校验。LLM 幻觉可能引入错误知识，且无主动通知机制触发人工审查。

### 4.2 MCP 协议层 — 最成熟的模块

[backend/app/mcp/protocol.py](file:///workspace/backend/app/mcp/protocol.py) 是整个 Agent Harness 中**最完整、最成熟**的模块。

**符合规范程度**：高。覆盖 MCP 2024-11-05 协议版本的 initialize/tools/resources/prompts 四大能力域，annotations 对齐 2025-06-18 规范。

**真正可用**：13 个工具全部有真实后端实现（不是 stub），调用 SearchEngine、RunbookGenerator、EventCorrelator、ChangeCorrelator、TopologyBuilder 等。

**13 个工具清单**：知识搜索、Runbook 生成、incident 管理（含状态机）、拓扑操作、影响分析等。每个工具有完整 `inputSchema`（JSON Schema），调用前运行时校验。

**6 个资源 URI**：`wiki://index`、`wiki://{slug}`、`topology://graph`、`documents://list`、`incidents://open`、`incidents://{id}`

**5 个 Prompt 模板**：`summarize_incident`、`generate_runbook_from_symptom`、`wiki_qa`、`root_cause_analysis`、`review_change_impact`，渲染时会拉取实际数据注入上下文。

**架构缺陷**：
- 同步 handler + async 后端的兼容性 hack（[protocol.py:977-1003](file:///workspace/backend/app/mcp/protocol.py)）：在已有事件循环运行时降级为空召回，是功能性 bug
- 无工具调用权限控制（所有 token 持有者可调用所有工具，含 `transition_incident` 改状态工具）
- 无工具调用审计日志（只记录错误，不记录正常调用的入参/出参/耗时）

### 4.3 流式响应

**Wiki Q&A 流式**（[wiki_query.py:689-767](file:///workspace/backend/app/knowledge/wiki_query.py)）：

SSE 四事件协议设计清晰：
```
event: meta   → {recalled_pages, cited_slugs, insufficient_knowledge, answer?}
event: delta  → {text: "chunk"}  （多次）
event: done   → {writebacks: [...]}
event: error  → {message: "..."}
```

**设计亮点**：
- meta 优先发送，前端在 LLM 生成期间立即展示引用来源
- 召回与生成共用 `_prepare_context()`，保证流式与非流式召回逻辑一致
- 流式结束后仍执行知识复利回写
- 设置 `X-Accel-Buffering: no` 禁用 nginx 缓冲

**关键差距**：**无中断/取消机制**。用户点击"停止生成"时，后端没有途径取消正在进行的 LLM stream 调用。`StreamingResponse` 客户端断开会触发 `asyncio.CancelledError`，但 `_stream_llm` 的 except 会静默吞掉，不传播到 LLM 客户端层。

### 4.4 AIOps 能力 — 真实算法

[backend/app/aiops/](file:///workspace/backend/app/aiops/) 5 个模块全部是**真实算法**，非 stub：

| 模块 | 算法 | 亮点 |
|------|------|------|
| `anomaly_detector.py` | Z-score + EWMA | 异常→事件桥接（detect_and_dispatch → webhook） |
| `change_correlator.py` | scope 重合分 + 时间临近分 + 变更类型权重 | suggest_rollback 回滚建议 |
| `event_correlator.py` | 指纹去重 + Union-Find 聚类 + 根因推断 | 完整 incident 状态机（open→ack→investigating→mitigated→resolved） |
| `timeseries_store.py` | SQLite + WAL 时序存储 | 滑动窗口算法 |
| `topology_builder.py` | 1535 行，共现推断 + 别名合并 + 快照 diff + 冗余度影响分析 | Mermaid/Cytoscape 导出 |

**AIOps 与 LLM 的协同**：AIOps 是纯算法，通过 MCP 层形成协同——LLM 角色是"分析者"（summarize_incident / root_cause_analysis prompt）而非"决策者"。所有自动化操作（状态迁移、回滚建议）都需人工通过 MCP 触发。

### 4.5 可观测性 — 生产级

[backend/app/observability/](file:///workspace/backend/app/observability/) 三件套齐全：

- **Prometheus 指标**：16 个业务指标 + HTTP 指标（Counter/Histogram/Gauge），独立 CollectorRegistry，`/metrics` 端点
- **OpenTelemetry 追踪**：W3C Trace Context 自动传播 + OTLP 导出 + 日志关联（trace_id/span_id 注入 structlog），span 覆盖 LLM/编译/关联/webhook 关键路径
- **业务指标采集**：后台任务每 30s 从存储层拉取数据更新 Gauge

**缺失**：LLM 指标有 `llm_calls_total` 和 `llm_call_duration_seconds`，但**缺少 token 用量指标**（prompt_tokens/completion_tokens 在 LLMResponse 中有但未上报 Prometheus）。作为 Agent Harness，token 成本监控是关键。

### 4.6 实时协作 — 生产级

[backend/app/realtime/](file:///workspace/backend/app/realtime/) WebSocket 实现完整：

- 多房间支持（按 slug 隔离）
- 编辑锁机制（acquire_lock/release_lock，软锁 + 乐观锁 + bypass_lock 应急通道）

- 事件持久化（5 类事件落 SQLite，游标分页）
- 4 个 Prometheus 指标埋点

## 六、后端 API 与存储审计

### 6.1 应用入口

[backend/app/main.py](file:///workspace/backend/app/main.py) 注册 25 个业务域 APIRouter，166 个端点。

**生命周期管理**：使用 `@asynccontextmanager lifespan`，启动时按序拉起 Webhook 重试 worker（15s）、可观测性采集器（30s）、协作 Hub 心跳清理、引导默认 admin 用户。关闭时取消所有任务。

**健康检查**：`GET /health`（liveness，始终 200）+ `GET /ready`（readiness，未就绪返回 503），对 K8s 友好。

### 6.2 API 设计一致性（需改进）

| 维度 | 评估 |
|------|------|
| 错误处理 | **不一致**。无全局 exception_handler；部分 `raise HTTPException`，部分 `except Exception: pass` 静默，部分返回 `{"error": "..."}` dict |
| 响应格式 | 多数返回 dict，少数用 response_model。无统一 envelope |
| 分页 | **不一致**。documents_router 用 limit+offset；llm_wiki_pages 仅 limit；realtime_router 用游标 before_id |
| 版本化 | **无**。无 `/api/v1` 前缀，破坏性变更无隔离 |
| CORS | **未配置**。生产必须依赖反向代理注入 CORS 头 |
| 鉴权 | 写操作普遍 `Depends(verify_token)`，读操作部分开放。`require_role` 分级 |

### 6.3 存储层瓶颈

6 个独立 SQLite DB 文件（documents/versions/webhooks/events/search/templates），WAL 模式。

**关键问题**：
- **横向扩展瓶颈**：SQLite 单文件无法安全跨实例共享，`deployment_mode=replicated` 需 NFS，但 SQLite over NFS 有已知锁损坏风险
- **连接管理不一致**：DocumentStore/VersionControl 不 `conn.close()`（泄漏风险），WebhookStore 规范关闭
- **VersionControl TOCTOU 竞态**：[version_control.py:74-111](file:///workspace/backend/app/storage/version_control.py) `save_version` 读 max_ver 后 INSERT 非原子，并发写可能版本号冲突
- **跨库事务非原子**：文档删除涉及 documents + search + topology + webhook 多库，无分布式事务

### 6.4 解析器覆盖度

注册格式 ≥25 种，主流办公文档（Office 全家桶）、网页、PDF、图片、数据格式（CSV/JSON/SQL）均有覆盖。

**风险**：MinerU 硬编码 pyenv 路径 `/root/.pyenv/versions/{ver}/bin/mineru`（[mineru_adapter.py:47-50](file:///workspace/backend/app/parsers/mineru_adapter.py)），非 root 环境失效。

## 七、基础设施与安全审计

### 7.1 CI/CD

**ci.yml**：backend job（ruff + pytest + 11 verify 脚本）+ frontend job（typecheck + vitest + build）。并发控制 `cancel-in-progress: true`，超时合理。

**docker-image.yml**：多架构（amd64+arm64）+ QEMU + buildx + GHA cache + provenance + sbom。冒烟测试设计良好（拉取镜像 → 轮询 healthcheck → 验证端点 → 失败输出诊断）。

**质量门禁缺口**：
1. **验证脚本覆盖率严重不足**（高）：40 个 verify 脚本仅 11 个进入 CI，29 个 s12-s17 脚本（SAML/LDAP/K8s/Dockerfile/前端路由/权限/测试覆盖/E2E/追踪 span/告警/异常/实时协作/OKF 适配等）未纳入 CI 门禁
2. **无依赖漏洞扫描**（无 pip-audit/safety/trivy/npm audit）
3. **无 SAST/镜像扫描**（无 CodeQL/Semgrep/Trivy 镜像扫描）
4. **无覆盖率门禁**（前端有 test:coverage 脚本但 CI 只跑 npm test）
5. **ruff 范围受限**（未覆盖 backend/tests/）

### 7.2 容器化

**Dockerfile** 多阶段构建（node:20-slim 前端 → python:3.12-slim 运行时），OCI LABEL 齐全，健康检查规范。

**严重安全问题**：
- **"非 root 运行"声明与事实不符**（严重）：Dockerfile 注释称"非 root 用户运行（安全）"，创建了 `opskg` 用户，但**全文件无 `USER` 指令**。`supervisord.conf` `[supervisord] user=root`，nginx master 以 root 启动（绑定 80 端口）。**容器主进程以 root 身份运行**，违背最小权限原则。
- **基础镜像未固定版本**：`python:3.12-slim` / `node:20-slim` 用滚动 tag，无 digest 锁定，供应链漂移风险。

### 7.3 Kubernetes 部署 — 不可直接部署

**结论：当前 K8s 清单不可直接部署，存在架构性不匹配。**

1. **【阻断】镜像架构不匹配**（严重）：Dockerfile 构建单镜像（前端 dist + 后端 + nginx + supervisord），但 K8s 清单假设前后端分离镜像（`opskg-backend:latest` + `opskg-frontend:latest`）。`opskg-frontend:latest` 从未被构建。
2. **【阻断】端口与监听地址不匹配**（严重）：`backend-deployment.yaml` 暴露 `containerPort: 8000`，三探针打 `port: http`(8000)。但单镜像内 uvicorn 绑定 `127.0.0.1:8000`，kubelet 从节点侧探测 pod_ip:8000 会失败。镜像真正对外的是 nginx:80，但 Deployment 未暴露 80。
3. **【阻断】ConfigMap checksum 占位**：`checksum/config: "placeholder"` 硬编码，ConfigMap 变更不会触发滚动更新。
4. **PVC 不可移植**：`storageClassName: nfs-client` 在多数集群不存在。
5. **SQLite + 多副本风险**：backend 2 副本共享 1 个 RWX PVC 跑 SQLite WAL，README 已警示并发写损坏风险但仍默认 2 副本。

**配置质量较好的部分**：资源 requests/limits 齐全、anti-affinity 软策略跨节点、三探针齐全、securityContext `drop: ["ALL"]`、NetworkPolicy 最小权限设计周全（default-deny + 精细规则 + `except` 排除内网段防 SSRF）、HPA 含 stabilizationWindow。

**配置缺失**：未设 `runAsNonRoot: true` / `runAsUser`、无 ResourceQuota/LimitRange/ServiceAccount/RoleBinding、Neo4j 单副本无 HA。

### 7.4 认证与授权 — 需加固

多认证方式（本地密码 + OIDC + SAML + LDAP）覆盖面广、设计对称，但安全实现存在严重缺陷：

| # | 问题 | 严重度 | 位置 |

| 1 | **密码哈希用 SHA256 而非 KDF** | 严重 | [models.py:70-75](file:///workspace/backend/app/auth/models.py) `SHA256(salt + password)`，无工作因子，GPU 暴力破解成本极低 |

## 十、总体结论

### 10.1 项目优势

OpsKG 在以下维度表现突出：

1. **Karpathy LLM Wiki 范式完整落地**：编译（LLM + 置信度门控 + 模板兜底）、知识复利（Q&A 回写）、漂移修正（checksum 检测 → 标记 → 重编译 → ReviewQueue 闭环）三要素齐备，且有完善的健康检查（7 类 Lint）+ OKF 互操作（三硬性约束 + 双轨链接 + 导入导出）+ 版本控制支撑。

2. **AIOps 真实算法引擎**：5 个模块全部是真实算法（Z-score/EWMA/Union-Find/共现推断/冗余分析），不是 stub，且有完整的 incident 状态机和告警→关联→知识闭环。

3. **MCP 协议实现成熟**：13 个工具 + 6 资源 + 5 Prompt 模板，全部有真实后端实现，JSON Schema 校验到位，符合 MCP 2024-11-05 + 2025-06-18 规范。

4. **可观测性生产级**：Prometheus 16 业务指标 + OpenTelemetry 自动 instrumentation + 日志关联 trace_id，span 覆盖 LLM/编译/关联/webhook 关键路径。

5. **实时协作生产级**：WebSocket + 编辑锁 + 心跳 + 连接上限 + 事件持久化 + 游标分页。

6. **前端工程现代化**：Vue 3.5 + Naive UI + Pinia 3 + TypeScript 6，菜单单一事实源落地，路由 meta 类型扩展，38 个 spec 覆盖广。

7. **防御性编程到位**：LLM 不可用兜底、permissive 降级、issue 忽略幂等、DOMPurify XSS 防护、LDAP RFC 转义。

### 10.2 项目短板

1. **安全实现存在严重缺陷**：弱密码哈希、OIDC 未验签、SSRF、logout 不失效、登录无限流、内存 state 破坏 HA。这些是生产前必须修复的阻断项。

2. **K8s 部署路径不可用**：清单与单镜像架构不匹配，按文档操作会失败。Docker 单镜像路径可用但以 root 运行。

3. **存储层单机瓶颈**：6 个独立 SQLite DB，无跨库事务，replicated 模式有锁损坏风险，横向扩展需迁移到 Postgres/MySQL。

4. **Agent 自主性不足**：本质是 RAG 而非 Agent。LLM 不在 agent loop 中自主选工具，无 function calling 原生支持，无重试/降级/限流。

5. **Wiki 与图谱割裂**：AGENTS.md 描述的"图谱增强 Wiki"未实现，Neo4j 仅被旧编译器使用，wiki_compiler/wiki_query 不消费图谱。

6. **CI 质量门禁不足**：40 个验证脚本仅 11 个进入 CI，29 个 s12-s17 验证无回归保护；无依赖扫描/SAST/覆盖率门禁。

### 10.3 最终定位

OpsKG 是一个 **"设计意图超前、核心范式落地、工程实现良好、安全加固待补"** 的项目。

- 作为 **LLM Wiki 知识管理平台**：**生产级可用**（安全加固后）
- 作为 **Agent Harness**：**可用但非自主**（RAG 架构，非 Agent 架构）
- 作为 **AIOps 平台**：**算法扎实**（5 个真实算法引擎）
- 作为 **生产部署产品**：**Docker 可用，K8s 待重写，安全待加固**

**建议优先级**：P0 安全加固 → P1 生产部署就绪 → P2 Agent Harness 增强 → P3 知识系统增强 → P4 工程质量。
