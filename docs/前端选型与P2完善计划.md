# OpsKG 前端选型与 P2 完善计划

> 生成日期: 2026-07-05 | 范围: 前端（P3 仅前端）+ P2 各模块完善

## 一、前端选型

### 1.1 选型背景

- **后端**: FastAPI（REST + MCP JSON-RPC），已完成 P0/P1/P2 共 50+ API 端点
- **使用场景**: 个人使用，中文界面，需图谱/拓扑可视化 + Markdown 编辑预览
- **关键依赖**: 图谱可视化、拓扑图、Markdown 编辑器、表格、表单
- **非目标**: SSR/SEO（运维内部工具，不需要搜索引擎收录）

### 1.2 候选方案对比

| 维度 | 方案 A: React 19 + Vite | 方案 B: Vue 3 + Vite (推荐) |
|------|------------------------|----------------------------|
| 生态规模 | 最大（42% 市场份额） | 大（28%，亚太 52%） |
| 上手速度 | 3-4 周（RSC 心智负担） | 2 周（SFC 直观） |
| 中文友好 | 中（shadcn/ui 英文为主） | 高（Naive UI 国人维护，中文文档） |
| Bundle 大小 | 45KB（React 19） | 34KB（Vue 3.4） |
| 拓扑可视化 | react-flow（成熟） | vue-flow（同等能力，基于 react-flow 移植） |
| Markdown 编辑 | @uiw/react-md-editor | md-editor-v3（国人维护，中文友好） |
| UI 组件库 | shadcn/ui（现代化） | Naive UI（完整 + 中文） |
| AI 工具适配 | 最佳（Cursor/Copilot） | 良好 |
| 性能 | 优秀 | 优秀（Vapor Mode 更快） |

### 1.3 推荐方案: Vue 3 + Vite + Naive UI

**技术栈**:
```
Vue 3.5 (Composition API + <script setup>)
Vite 6 (构建)
Vue Router 4 (路由)
Pinia 3 (状态管理)
Naive UI (组件库，中文友好)
vue-flow (拓扑/图谱可视化)
md-editor-v3 (Markdown 编辑预览)
axios (HTTP 客户端)
@vueuse/core (工具集)
Tailwind CSS 4 (原子化 CSS，可选)
```

**选型理由**:
1. **中文生态友好** — Naive UI / md-editor-v3 均国人维护，文档中文完整
2. **上手快** — 个人使用，2 周可产出 MVP；SFC 结构直观
3. **性能优** — Vue 3.4 bundle 34KB，Core Web Vitals 优于 React
4. **拓扑可视化成熟** — vue-flow 与 react-flow 同源，能力对等
5. **后端解耦** — 纯 SPA，通过 REST/MCP 调用后端，部署简单

**备选**: 若偏好 React 生态或未来考虑团队协作，可选方案 A（React 19 + shadcn/ui + react-flow）。

### 1.4 前端能力计划（14 个模块）

按优先级分 3 批交付：

#### 第一批: 核心闭环（MVP，覆盖 P0/P1 主要 API）

| # | 模块 | 对接 API | 关键交互 |
|---|------|----------|----------|
| F1 | 仪表盘 | `/health` `/documents/stats` `/review/stats` `/search/stats` | 总览卡片 + 最近活动 |
| F2 | 文档管理 | `/documents` `/parsers/parse/*` | 上传(拖拽) + 列表 + 解析结果查看 + 删除 |
| F3 | 知识搜索 | `/search` `/documents/{id}` | 搜索框 + 结果高亮 + 文档详情抽屉 |
| F4 | 审查队列 | `/review/queue` `/review/{id}/approve\|reject` `/review/batch-approve` | 列表 + 批量操作 + 详情面板 |
| F5 | Wiki 浏览 | `/wiki` `/wiki/{slug}` `/versions/wiki/{slug}/*` | 页面树 + 内容渲染 + 版本历史 + diff |

#### 第二批: AIOps 运维场景（覆盖 P2 主要 API）

| # | 模块 | 对接 API | 关键交互 |
|---|------|----------|----------|
| F6 | Runbook 工作台 | `/runbook/generate` `/runbook/preview` `/templates/*/render` | 表单(症状/服务/主机) + 预览 + 发布 Wiki |
| F7 | Incident 管理 | `/events/incidents` `/events/incidents/{id}` `/events/incidents/{id}/close` `/events/incidents/{id}/runbook` | 列表 + 详情(告警时间线) + 关闭 + 一键生成 Runbook |
| F8 | 变更关联 | `/changes` `/changes/correlate` `/events/incidents/{id}/changes` `/events/incidents/{id}/rollback-suggestion` | 变更列表 + 关联矩阵 + 回滚建议卡片 |
| F9 | 服务拓扑 | `/topology` `/topology/nodes/{name}` `/topology/impact/{name}` | vue-flow 力导向图 + 节点点击 + 影响分析侧栏 |

#### 第三批: 增强能力

| # | 模块 | 对接 API | 关键交互 |
|---|------|----------|----------|
| F10 | 模板管理 | `/templates` `/templates/{slug}` | CRUD + 变量预览 + 渲染测试 |
| F11 | 版本控制 | `/versions/{key}/*` | 版本时间线 + diff 视图 + 回滚 |
| F12 | 导出中心 | `/export` | 多格式下载（MD/HTML/TXT/PDF） |
| F13 | MCP 工具浏览器 | `/mcp/tools` `/mcp` | 工具列表 + 参数表单 + 试调用 + 结果展示 |
| F14 | 知识图谱可视化 | `/graph/*` | Neo4j 可选，图谱力导向图（Neo4j 未连接时降级提示） |

## 二、P2 各模块完善计划

基于代码盘点，按"工程化不足 → 算法粗糙 → 集成缺失"排序。

### 2.1 P2-1 Runbook 自动生成器

**当前**: FTS5 关键词检索 + 规则抽取 + 固定模板，无反馈闭环

| 编号 | 任务 | 优先级 | 验收标准 |
|------|------|--------|----------|
| P2-1.1 | **向量检索 + RRF 融合** | 高 | hybrid score = 0.4*keyword + 0.6*vector，召回率提升 ≥30% |
| P2-1.2 | **文档实体抽取缓存** | 高 | 缓存 key=doc_id+checksum，二次生成耗时降 ≥50% |
| P2-1.3 | **Runbook 持久化 + 版本** | 高 | 自动发布到 wiki:runbook-* 并建立版本，支持编辑迭代 |
| P2-1.4 | **采纳/反馈闭环** | 中 | 记录"采纳/否决/编辑"反馈表，用于后续优化排序 |
| P2-1.5 | **中文分词优化** | 中 | jieba 分词，改善中文 query 召回 |
| P2-1.6 | **空结果兜底** | 低 | 检索 0 命中时提示"知识库不足"并建议上传相关文档 |

### 2.2 P2-2 事件关联引擎

**当前**: Union-Find 聚类 + 启发式根因，无去噪，状态机只有 open/closed

| 编号 | 任务 | 优先级 | 验收标准 |
|------|------|--------|----------|
| P2-2.1 | **告警指纹去重 + 抑制** | 高 | 同 fingerprint 5 分钟内去重；同 host 同 message 抑制 |
| P2-2.2 | **incident 状态机** | 高 | open→acknowledged→investigating→mitigated→resolved，含状态流转约束 |
| P2-2.3 | **修复 noise_filtered 恒 0** | 高 | 真正过滤单事件 noise（合并到相邻 incident 或标记为 noise） |
| P2-2.4 | **基于真实拓扑做根因推断** | 中 | 调用 topology_builder 获取依赖图，上游节点先告警→根因候选 |
| P2-2.5 | **Alertmanager webhook 接入** | 中 | `/events/webhook/alertmanager` 接收标准 Alertmanager payload |
| P2-2.6 | **跨窗口合并优化** | 低 | scope 子集包含即合并，避免碎片化 |

### 2.3 P2-3 变更关联引擎

**当前**: 三维评分 + 回滚建议，但 type_weight 权重稀释严重（仅贡献 0.04-0.09）

| 编号 | 任务 | 优先级 | 验收标准 |
|------|------|--------|----------|
| P2-3.1 | **修复 type_weight 权重稀释** | 高 | 调整评分公式，type_weight 直接贡献（非 *0.1），区分度提升 |
| P2-3.2 | **关联算法索引预过滤** | 高 | 按 host/service 建索引，避免 N×M 双重遍历 |
| P2-3.3 | **变更内容 diff 分析** | 中 | config_change 解析 key 变更；deployment 记录镜像/commit |
| P2-3.4 | **利用 rollback_of 字段** | 中 | 识别回滚变更，降低其作为根因的权重 |
| P2-3.5 | **时间窗口可配 + 长尾支持** | 中 | 默认 30min，deployment 类支持 2h 长尾窗口 |
| P2-3.6 | **CMDB 实体归一化** | 低 | 别名/IP↔主机名映射，提升 scope 重合度准确性 |
| P2-3.7 | **回滚执行链路** | 低 | 对接发布系统 API（ArgoCD/Jenkins），回写结果 |

### 2.4 P2-4 服务拓扑自动构建

**当前**: 全连接推断（service×host + service×component），假阳性边多

| 编号 | 任务 | 优先级 | 验收标准 |
|------|------|--------|----------|
| P2-4.1 | **关系推断改为上下文共现** | 高 | 基于 evidence_span 同段落/同列表共现 + 置信度评分，剔除全连接假阳性 |
| P2-4.2 | **节点别名合并** | 高 | db1.example.com ↔ db1 合并；支持别名映射表 |
| P2-4.3 | **拓扑 diff 检测** | 中 | snapshot 对比，输出新增/消失/变更边，触发 topology_changed 事件 |
| P2-4.4 | **Mermaid/Cytoscape 导出** | 中 | `/topology/export?format=mermaid\|cytoscape` 供前端/文档消费 |
| P2-4.5 | **文档删除时清理拓扑** | 中 | 文档删除触发节点/边 occurrences 递减，归零时清除 |
| P2-4.6 | **节点 metadata 扩展** | 低 | 增加 IP/版本/owner/环境/region 字段 |
| P2-4.7 | **影响分析结合冗余度** | 低 | 集群多副本时单节点故障不算影响（需容量元数据） |

### 2.5 P2-5 MCP 协议支持

**当前**: 8 工具 + JSON-RPC，但协议版本旧，缺 resources/prompts/SSE/auth

| 编号 | 任务 | 优先级 | 验收标准 |
|------|------|--------|----------|
| P2-5.1 | **升级协议版本 + 补全 capabilities** | 高 | 升级到 2025-06-18，声明 resources/prompts/logging 能力 |
| P2-5.2 | **token auth** | 高 | /mcp 端点走 verify_token（与 REST 一致） |
| P2-5.3 | **resources 实现** | 高 | 知识库文档暴露为 resources（uri=doc://{doc_id}） |
| P2-5.4 | **prompts 实现** | 中 | Runbook 模板暴露为 prompts，支持参数填充 |
| P2-5.5 | **SSE 传输 + progress 通知** | 中 | long-running 工具（generate_runbook）推送进度 |
| P2-5.6 | **batch 请求支持** | 中 | handle_request 支持数组批量处理 |
| P2-5.7 | **工具 annotations** | 中 | 标注 readOnlyHint/destructiveHint/idempotentHint |
| P2-5.8 | **JSON Schema 运行时校验** | 低 | 入参校验，缺失字段返回 -32602 Invalid Params |
| P2-5.9 | **list 工具分页** | 低 | get_topology 等支持 cursor/nextCursor 分页 |

## 三、执行顺序建议

### 阶段 1: P2 高优先级完善（与前端并行启动）

按"修复 bug > 工程化 > 算法优化"排序：

1. **P2-2.3** 修复 noise_filtered 恒 0（bug）
2. **P2-3.1** 修复 type_weight 权重稀释（bug）
3. **P2-1.2** 文档实体抽取缓存（性能）
4. **P2-4.1** 关系推断改为上下文共现（算法质量）
5. **P2-2.1** 告警指纹去重 + 抑制（工程化）
6. **P2-2.2** incident 状态机（工程化）
7. **P2-5.1 + P2-5.2** MCP 协议升级 + auth（协议基础）

### 阶段 2: 前端 MVP（第 1-2 周）

F1-F5 核心闭环，与 P2 完善并行。

### 阶段 3: P2 中优先级 + 前端 AIOps（第 3-4 周）

P2-1.1 向量检索、P2-2.4 拓扑根因、P2-4.3 diff 检测；前端 F6-F9。

### 阶段 4: P2 低优先级 + 前端增强（第 5 周+）

P2-3.6 CMDB、P2-5.5 SSE、P2-4.7 冗余度；前端 F10-F14。
