## 二、专项一：流水线处理审计

### 2.1 现状分析

#### 2.1.1 当前"流水线"实际架构

项目当前**没有统一的流水线编排系统**。文档处理流程分散在多个 API 端点和独立模块中：

```
当前实际流程（非统一编排）:

┌─────────────────────────────────────────────────────────────────────┐
│  POST /parsers/parse/{fmt}         POST /llm-wiki/ingest            │
│  ┌──────────────────────┐         ┌──────────────────────┐         │
│  │ 1. 上传文件           │         │ 1. 上传文件           │         │
│  │ 2. 保存到磁盘         │         │ 2. 保存到磁盘         │         │
│  │ 3. 解析文档           │         │ 3. 漂移检测           │         │
│  │ 4. 更新状态→parsed    │         │ 4. 解析文档           │         │
│  │ 5. 建立搜索索引       │         │ 5. 知识抽取(LLM)      │         │
│  └──────────────────────┘         │ 6. Wiki编译(LLM)       │         │
│           ↓                        │ 7. 写回VersionControl   │         │
│   前端点击"编译为Wiki"              │ 8. 更新backlink        │         │
│           ↓                        │ 9. 重建index           │         │
│  POST /llm-wiki/recompile/{id}     └──────────────────────┘         │
│  (重新执行 parse+extract+compile)                                    │
└─────────────────────────────────────────────────────────────────────┘
```

两个入口各自独立，互不感知：
- `/parsers/parse/{fmt}` — 只做解析，不触发编译（[parsers_router.py:142-197](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L142-L197)）
- `/llm-wiki/ingest` — 重新上传+解析+抽取+编译，全量重做（[llm_wiki_router.py:104-156](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/llm_wiki_router.py#L104-L156)）
- 前端"编译为Wiki"按钮 — 调用 `/llm-wiki/recompile/{doc_id}`，重新 parse+extract+compile（[wiki.ts:215-219](file:///d:/workfiles/only-LLMwiki-comp/frontend/src/api/wiki.ts#L215-L219)）

#### 2.1.2 状态追踪粒度

DocumentStore 的文档状态机（[document_store.py:51](file:///d:/workfiles/only-LLMwiki-comp/backend/app/storage/document_store.py#L51)）：

定义了 5 个状态，但实际代码中：

- `error` 状态在解析失败时设置（[parsers_router.py:136](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L136)）

### 2.2 问题清单

#### 问题 1：无统一流水线编排（严重）

**现状**：文档处理流程（上传→解析→抽取→编译→建索引→回链）分散在 3 个独立 API 端点中，无统一编排层。

**代码证据**：
- 解析端点：[parsers_router.py:71-139](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L71-L139) — 批量解析用 for 循环串行处理，无并发
- Ingest 端点：[llm_wiki_router.py:104-156](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/llm_wiki_router.py#L104-L156) — 重新上传文件，重复 parse
- 前端重编译：[wiki.ts:215-219](file:///d:/workfiles/only-LLMwiki-comp/frontend/src/api/wiki.ts#L215-L219) — 调用 recompile，重新 parse+extract+compile

**影响**：
- 用户无法看到"解析完成→抽取中→编译中→索引重建中"等步骤级进度
- 同一个文档被 parse 多次（先 /parse 再 /ingest 或 /recompile）
- 无法追踪流水线执行历史（哪个步骤成功、哪个失败、耗时多少）

#### 问题 2：前端无步骤级可视化（严重）

**现状**：前端 DocumentsView 仅显示一个简单的状态标签（"已上传"/"已解析"/"失败"），无任何步骤级进度展示。

**代码证据**：[DocumentsView.vue:100-111](file:///d:/workfiles/only-LLMwiki-comp/frontend/src/views/DocumentsView.vue#L100-L111) — 状态列仅渲染 NTag：

```typescript
render(row: DocumentMeta) {
  return h(NTag, { type: statusTagType[row.status], size: 'small' },
    { default: () => statusText[row.status] },
  )
},
```

**缺失的能力**：
- 无流水线步骤可视化（如 Step 1/4: 解析中 → Step 2/4: 抽取中 → ...）
- 无实时进度反馈（当前是同步 HTTP 请求，等待整个编译完成才返回）
- 无步骤耗时展示
- 无步骤日志/错误详情展示

#### 问题 3：无断点重跑能力（严重）

**现状**：当某个步骤失败时，唯一的"重试"方式是重新调用整个流程。

**代码证据**：
- 解析失败 → 需要重新上传文件（[parsers_router.py:135-137](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L135-L137)）
- 编译失败 → 前端调用 `/llm-wiki/recompile/{doc_id}`，重新执行 parse+extract+compile 全流程（[wiki_compiler.py:181-331](file:///d:/workfiles/only-LLMwiki-comp/backend/app/knowledge/wiki_compiler.py#L181-L331)）
- 无中间状态持久化 — 解析结果只在内存中，重跑时重新解析

**具体场景**：
1. 用户上传一个 50 页的 Word 文档
2. 解析成功（耗时 2s），但 LLM 编译时因 API 超时失败
3. 用户点击"重试" → 重新上传、重新解析、重新编译
4. 如果 LLM 再次超时，又得从头再来

**缺失的能力**：
- 无法从失败步骤继续（如"编译失败，从编译步骤重试"）
- 无法跳过已成功的步骤
- 无幂等性保证（重复执行产生重复 wiki 页面）

#### 问题 4：批量解析无进度反馈

**现状**：[parsers_router.py:71-139](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L71-L139) 的批量解析是同步 HTTP 请求，所有文件在 for 循环中串行处理完毕后一次性返回结果。

**影响**：上传 10 个文件时，用户等待所有文件解析完才能看到结果。中途某个文件失败，不知道是哪个。

#### 问题 5：前端/后端状态值不一致

| 前端状态值 | 后端状态值 | 前端筛选器 |
|-----------|-----------|-----------|
| `uploaded` | `uploaded` | 可用 |
| `parsing` | **不存在** | 永远匹配不到 |
| `parsed` | `parsed` | 可用 |
| `failed` | `error` | 永远匹配不到 |
| — | `extracted` | **未暴露** |
| — | `compiled` | **未暴露** |

### 2.3 改进建议

```
建议架构（统一流水线编排）:

POST /pipeline/run/{doc_id}
  → 返回 pipeline_run_id（UUID）
  → SSE 推送步骤进度事件

SSE 事件流:
  event: step_start  → {step: "parse", pipeline_run_id: "..."}
  event: step_progress → {step: "parse", progress: 0.5, message: "正在解析第3页..."}
  event: step_done   → {step: "parse", duration_ms: 2300, result: {...}}
  event: step_error  → {step: "extract", error: "LLM 超时", retryable: true}
  event: pipeline_done → {total_duration_ms: 45000, pages_created: 3}

POST /pipeline/retry/{pipeline_run_id}?from_step=extract
  → 从 extract 步骤重新执行（跳过 parse）
```

**关键实现要素**：
1. **PipelineRun 状态机**：记录每个步骤的状态（pending/running/done/error）、开始时间、结束时间、输出
2. **SSE 实时推送**：前端通过 EventSource 订阅流水线进度
3. **断点续跑**：每个步骤的输出持久化，retry 时复用已成功的步骤结果
4. **步骤级幂等**：每个步骤有独立 checksum，相同输入+相同步骤=跳过
5. **前端步骤可视化**：Stepper 组件展示流水线步骤，实时高亮当前步骤，展示耗时和状态

## 四、专项三：文档解析审计

### 4.1 现状分析

#### 4.1.1 解析器架构

项目注册了 25+ 种格式的解析器（[registry.py:61-138](file:///d:/workfiles/only-LLMwiki-comp/backend/app/parsers/registry.py#L61-L138)）：

| 解析器 | 覆盖格式 | 策略 |
|--------|---------|------|
| MarkItDown | docx/doc/xlsx/xls/pptx/ppt/html/htm/epub/csv/json/pdf(fallback) | 主力，MIT 许可，纯 Python |
| MinerU | pdf/xlsx/xls/png/jpg/jpeg/gif/bmp | PDF/Excel 专项，VLM+OCR |
| 自研 MarkdownParser | md/markdown | 结构提取 |
| 自研 SQLParser | sql | DDL/DML 语义识别 |
| 自研 TextParser | txt | 简单分段 |
| Unstructured | rst/xml/odt/msg/eml | 备选 fallback |

解析流程：原始文件 → 解析器 → Markdown → MarkdownParser → ParsedDocument（结构化元素列表）

#### 4.1.2 解析结果管道截断问题（核心问题）

这是文档解析"与原文差别很大、完全不可用"的**根本原因**。整个管道存在三级截断：

**第一级：MarkdownParser 解析截断**

MarkdownParser 解析整个 Markdown 文本，生成完整的 `ParsedElement` 列表。这个阶段通常是完整的。

**第二级：API 响应截断（50 元素）**

[parsers_router.py:46-63](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L46-L63)：

```python
def _serialize_doc(doc: ParsedDocument) -> dict:
    return {
        ...
        "elements": [
            { ... }
            for e in doc.elements[:50]  # ← API 响应限制前 50 个元素
        ],
    }
```

API 返回的解析结果只包含前 50 个元素。一份 100 页的 Word 文档可能产生 500+ 个元素，API 只返回前 10%。

**第三级：知识抽取截断（20 元素 + 500 字符/元素）**

[extractor.py:107-116](file:///d:/workfiles/only-LLMwiki-comp/backend/app/extraction/extractor.py#L107-L116)：

```python
def _build_context(self, doc: ParsedDocument) -> str:
    lines = [f"# {doc.title or doc.doc_id}", ""]
    for e in doc.elements[:20]:          # ← 只用前 20 个元素
        label = f"[{e.type.value}]"
        if e.section:
            label += f"（章节：{e.section}）"
        lines.append(f"{label} {e.content[:500]}")  # ← 每个元素截断到 500 字符
        lines.append("")
    return "\n".join(lines)
```

知识抽取（LLM 调用的输入）只使用前 20 个元素，每个元素内容截断到 500 字符。这意味着：
- 一份 100 页的 Word 文档，LLM 实际只看到前 20 段（约前 2-3 页）的内容
- 第 3 页之后的所有内容在知识抽取阶段就被丢弃了

**第四级：Wiki 编译 evidence_span 截断（1200 字符）**

### 4.2 问题清单

#### 问题 1：文档内容查看完全不可用（严重）

**现象**：对所有二进制格式（docx/xlsx/pdf/pptx），前端"查看"显示 `[二进制内容，共 N 字节，无法预览]`。

**根因**：[documents_router.py:77-80](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/documents_router.py#L77-L80) 返回原始文件字节，未返回解析后的文本。

**修复方向**：应返回解析后的 Markdown 文本（读取 `parse_result` 字段或重新解析）。

#### 问题 2：解析结果管道截断导致大部分内容丢失（严重）

**三级截断汇总**：

| 截断点 | 位置 | 截断量 | 影响 |
|--------|------|--------|------|
| API 响应 | [parsers_router.py:61](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L61) | 仅返回前 50/500+ 个元素 | 后端 API 消费者只能看到片段 |
| 知识抽取 | [extractor.py:109](file:///d:/workfiles/only-LLMwiki-comp/backend/app/extraction/extractor.py#L109) | 仅用前 20 个元素 | LLM 只看到前 2-3 页 |
| 元素内容 | [extractor.py:114](file:///d:/workfiles/only-LLMwiki-comp/backend/app/extraction/extractor.py#L114) | 每元素截断到 500 字符 | 长段落/大表格被截断 |
| 证据片段 | [wiki_compiler.py:474](file:///d:/workfiles/only-LLMwiki-comp/backend/app/knowledge/wiki_compiler.py#L474) | 截断到 1200 字符 | 多段落证据丢失 |

**根因**：所有截断的硬编码限制是为了控制 LLM token 消耗，但限制过于激进。

**实际效果**：假设上传一份 50 页的 Word 运维手册（约 500 个元素），LLM 实际只看到前 20 个元素 × 500 字符 = 最多 10,000 字符（约 2,500 tokens）。这意味着手册 95% 的内容在知识库中完全消失。

#### 问题 3：解析结果未存储/未展示

**现象**：解析后的结构化数据（ParsedDocument）在 API 响应中返回后即丢弃，未持久化到数据库。

**代码证据**：[parsers_router.py:118](file:///d:/workfiles/only-LLMwiki-comp/backend/app/routers/parsers_router.py#L118)：
```python
store.update_status(doc_meta["doc_id"], "parsed", title=doc.title)
# 未传 parse_result=_serialize_doc(doc)
```

**影响**：
- 前端无法展示解析后的结构化内容（表格、标题、代码块等）
- 每次重编译都需要重新解析（浪费计算资源）
- 无法对比"原始文档 vs 解析结果"以验证解析质量

#### 问题 4：MarkItDown 链式转换导致质量损失

**流程**：原始文件 → MarkItDown → Markdown → MarkdownParser → ParsedDocument

**问题**：
- MarkItDown 转换 Word/Excel 时已损失格式信息（如代码块丢失、合并单元格信息丢失）
- MarkdownParser 再解析时，对 MarkItDown 生成的 Markdown 做二次解析，可能进一步损失信息
- 表格中的 `Unnamed: X` 列名会在后续编译中产生无意义的 wiki 实体

#### 问题 5：前端未展示解析结果

**现状**：DocumentsView 的"查看"抽屉只展示原始内容（二进制文档显示为不可预览），**不展示解析后的结构化元素**。

**缺失的展示**：
- 解析后的元素列表（标题/段落/表格/代码块/列表）
- 元素类型区分（不同颜色/图标标识）
- 表格渲染（Markdown 表格 → HTML 表格）
- 代码块高亮
- 解析结果与原始文档的并排对比

```python
# extractor.py 改进
def _build_context(self, doc: ParsedDocument) -> str:
    """构建 LLM 上下文：分块处理，每块最多 20 个元素，多次 LLM 调用"""
    chunks = []
    chunk_size = 20
    for i in range(0, len(doc.elements), chunk_size):
        chunk = doc.elements[i:i + chunk_size]
        chunk_text = self._render_chunk(chunk)
        chunks.append(chunk_text)
    
    # 如果元素数 <= 20，直接返回
    if len(chunks) <= 1:
        return chunks[0] if chunks else ""
    
    # 多块：合并所有块的抽取结果
    # 每块单独调 LLM，最终合并实体/关系
    return chunks  # 调用方需要改为迭代处理
```
