# OpsKG 项目可用性与使用情况审计报告

**审计日期：** 2026-07-16  
**项目版本：** v0.0.1  
**审计范围：** 全栈（前端/后端/部署/安全/质量）  
**审计角度：** 专业用户（运维工程师/知识管理员）

---

## 一、项目概览

### 1.1 定位与架构

OpsKG 是一个 **LLM 驱动的运维知识图谱 Wiki 控制台**，基于 Karpathy LLM Wiki 范式，将原始运维文档（L1 Raw Layer）编译为结构化 Markdown Wiki（L2 Wiki Layer），通过 AGENTS.md 规范（L3 Schema Layer）治理知识网络。

**技术架构：**

| 层 | 技术栈 | 规模 |
|---|---|---|
| 前端 | Vue 3 + TypeScript + Naive UI + Vite | 142 文件（24 视图, 8 stores, 10 composables） |
| 后端 | Python 3.12 + FastAPI + SQLite/Neo4j | 158 文件（28 路由器, ~7778 行路由代码） |
| 部署 | Docker + Nginx + Supervisord | 多阶段 Dockerfile, 非 root 用户, 单镜像交付 |
| 质量 | pytest + vitest + ruff + vue-tsc | 498 后端测试, 51 前端 spec, 40 verify 脚本 |
| CI | GitHub Actions | 双作业（backend + frontend），含 pip-audit 安全扫描 |

### 1.2 代码规模

- 后端 Python：158 文件，路由器约 7,778 行，核心模块约 3 万行
- 前端 Vue/TS：142 文件，约 3.2 万行
- 测试文件：后端 41 测试文件，前端 51 spec 文件
- 验证脚本：40 个 verify 脚本，全部纳入 CI 门禁
- 配置文件：config.py 286 行，含 ~60 个可配置参数

## 二、前端可用性审计

### 2.1 路由与导航结构

**评分：优秀（90/100）**

路由系统以 router/index.ts 为单一事实源，共 22 个路由，按 5 个功能分组组织：

| 菜单组 | 页面数 | 包含路由 |
|---|---|---|
| 仪表盘 | 1 | `/dashboard` |
| 知识管理 | 6 | 文档管理、知识搜索、Wiki 浏览、Wiki Q&A、知识图谱、编译流水线 |
| 质量治理 | 4 | 健康检查、管道追踪、审查队列、版本控制 |
| AIOps | 4 | Incident 管理、变更关联、服务拓扑、Runbook 工作台 |
| 系统工具 | 5 | 模板管理、导出中心、MCP 浏览器、用户管理(admin)、系统配置(admin) |

**优点：**
- 菜单从路由 meta 自动派生（`buildMenuOptions`），避免硬编码，菜单与路由始终保持一致
- 角色过滤与路由守卫一致，admin/operator/viewer 三级权限，`requireRole` 在路由层拦截
- 路由级 loading bar（`startLoadingBar`/`finishLoadingBar`），含 chunk 下载反馈
- 最近访问页面动态显示在侧栏顶部，提升高频使用效率
- 404 兜底和 403 禁止访问页面，用户体验完整
- 公开路由（登录/Setup/回调/403）不在 AppLayout 内，避免侧栏干扰

**待改进：**
- 缺少面包屑导航（breadcrumb），深层页面返回不便
- 缺少页面级快捷键支持
- 无搜索框全局搜索页面功能（侧栏仅有菜单，无搜索入口）

### 2.2 页面覆盖度

**评分：良好（82/100）**

24 个 Vue 视图全部有对应的 spec 文件（vitest 单元测试），覆盖度良好。但存在两个例外：

- PipelineView.vue — 编译流水线核心页面，无 spec 测试
- PipelineTraceView.vue — 管道追踪，无 spec 测试

这两个页面是知识编译的核心交互界面，缺少测试存在回归风险。

### 2.3 状态管理

**评分：良好（85/100）**

使用 Pinia 管理 8 个 store，覆盖核心业务状态：

| Store | 用途 |
|---|---|
| `app` | 全局 UI 状态（侧栏折叠等） |
| `auth` | 认证状态（用户信息、角色、authRequired） |
| `setup` | 开箱配置向导状态 |
| `onboarding` | 新手引导状态 |
| 其他 | 各业务域状态 |

**优点：** 认证状态与路由守卫联动，`authRequired` 三态（true/false/null）区分"需认证/dev模式/后端不可达"。

**待改进：** 缺少全局错误状态 store，当前错误处理分散在各组件中。

### 2.4 组件体系

**评分：良好（83/100）**

通用组件覆盖了基础 UI 场景：
- `EmptyState` — 空状态占位
- `ErrorBoundary` — 错误边界
- `LoadingState` / `PageSkeleton` — 加载态
- `PageHeader` — 页面标题
- `CollabPanel` / `WikiEditor` / `WikiVersionHistory` — 协作与 Wiki 编辑

**待改进：** 缺少 Toast/Notification 全局通知组件、确认对话框（ConfirmDialog）组件、表格/列表通用组件。

### 2.5 Composables 复用

**评分：良好（80/100）**

10 个 composables 覆盖了关键可复用逻辑：
- `useCollab` / `useCollabHistory` — 实时协作
- `useEditDraft` — 编辑草稿
- `usePermission` — 权限检查
- `useRecentPages` — 最近访问
- `useSse` — Server-Sent Events 流式通信

**待改进：** 缺少 `useDebounce`、`usePagination`、`useForm` 等通用 composable。

## 三、后端能力审计

### 3.1 API 端点覆盖

**评分：优秀（92/100）**

main.py 注册了 22 个业务路由器 + 6 个基础设施路由器，采用 `/api/v1` 版本化 + `/api` 兼容 + 无前缀直连的三层兼容策略。

基础设施端点（不版本化）：
- `/auth/*` — 登录/登出/改密/用户CRUD/角色列表
- `/auth/oidc/*` — OIDC SSO 多提供者
- `/auth/saml/*` — SAML 2.0 SSO
- `/auth/ldap/*` — LDAP/AD 认证
- `/setup/*` — 开箱配置向导
- `/health` / `/ready` — K8s 探针
- `/metrics` — Prometheus 指标
- `/api/audit/*` — 审计日志查询

**优点：**
- 完整的 RESTful 设计，CRUD 操作语义清晰
- 基础设施端点不版本化，业务端点版本化，设计合理
- Request ID 追踪，X-Request-ID 头支持
- 全局异常处理器，生产环境不泄露堆栈

### 3.2 认证与授权

**评分：优秀（93/100）**

支持四种认证方式，覆盖从小团队到企业级场景：

| 方式 | 状态 | 说明 |
|---|---|---|
| 用户名密码 | 完整 | bcrypt 哈希，5次/分钟限流，账户锁定，强制改密 |
| API Token | 完整 | Bearer token，dev 模式留空放行 |
| OIDC SSO | 完整 | 多提供者，nonce 防重放，PKCE，自动创建用户 |
| SAML 2.0 | 完整 | 严格模式签名验证，自动创建用户 |
| LDAP/AD | 完整 | 企业目录集成，自动创建用户 |

**安全加固点：**
- 密码 bcrypt 哈希存储
- 登录 5次/分钟/IP 限流（防暴力破解）
- OIDC nonce 验证（防重放攻击）
- Token 撤销（服务端删除 session）
- 账户锁定机制
- 新用户强制改密标志

### 3.3 LLM 集成

**评分：优秀（90/100）**

支持三种 LLM 后端，通过 resilient.py 提供弹性调用：

| 后端 | 适用场景 |
|---|---|
| `openai_compat` | DeepSeek/通义/OpenAI，云端 API |
| `ollama` | 本地部署，免费 |
| `vllm` | 本地 GPU 推理，高性能 |

**弹性能力：**
- 指数退避重试（最多 3 次，区分可重试/不可重试错误）
- 并发限流（Semaphore，默认 10）
- Token Bucket 限流（可配置 req/s）
- 降级链（主后端失败 → fallback 后端链）
- Token 用量上报 Prometheus
- 流式响应支持取消（CancelToken）

**待改进：** 缺少 LLM 调用缓存（相同 prompt 短期内复用结果），缺少 LLM 成本统计 dashboard。

### 3.4 知识编译流水线

**评分：优秀（88/100）**

核心能力对标 Karpathy LLM Wiki 范式，wiki_compiler.py 实现了完整的三层架构：

**工作流：**
```
DocumentStore (raw) → KnowledgeExtractor → WikiCompiler → VersionControl (wiki:*)
                                    ↓
                              GraphStore (辅助索引)
```

**关键特性：**
- 增量编译：raw 更新 → 合并新事实 → 标注 stale → 触发重编译
- 6 种页面类型（entity/concept/incident/runbook/service/host），各有必含章节骨架
- `[[wikilink]]` 双向链接，自动维护反向索引
- SSE 进度推送（前端实时展示编译进度）
- 质量评分（≥0.8 自动发布，≥0.5 review_needed，<0.5 拒绝）
- 章节级重编译 + LLM 参数调整
- 知识复利回写（LLM 自校验防幻觉）

### 3.5 Wiki 健康检查

**评分：优秀（87/100）**

wiki_lint.py 实现 6 类检测：

| 检测类型 | 说明 |
|---|---|
| 矛盾检测 | 同一概念在多个页面有冲突描述（regex + LLM 语义） |
| Stale 检测 | raw 文档变更但 wiki 未重编译 |
| Orphan 检测 | wiki 页面无入链 |
| 死链检测 | `[[slug]]` 指向不存在的页面 |
| 缺失概念检测 | 图谱实体无 wiki 页面 |
| OKF 合规检测 | OKF v0.1 格式约束校验 |

### 3.6 可观测性

**评分：良好（84/100）**

三层可观测性体系：
- **Metrics**：Prometheus 指标（LLM token 用量、业务指标采集器）
- **Tracing**：OpenTelemetry 分布式追踪（可选启用）
- **Logging**：structlog 结构化日志 + 独立审计日志

**审计日志**（audit_log.py）：
- 仅记录写操作（POST/PUT/PATCH/DELETE）
- 敏感字段自动脱敏（password/token/secret/api_key）
- 异步写入 SQLite（不阻塞业务）
- 支持按用户/方法/路径/时间过滤查询

**待改进：** 缺少预设 Grafana Dashboard JSON，缺少告警规则模板。

### 5.3 API 版本化

**评分：良好（82/100）**

- `/api/v1` — 主版本

**待改进：** 缺少 API 版本废弃策略（deprecation header 通知），无 API 变更日志（CHANGELOG）。

## 九、结论

OpsKG 是一个**架构设计优秀、安全防护到位、功能覆盖全面**的 LLM 运维知识管理平台。其核心亮点是：

1. **Karpathy LLM Wiki 范式**的完整实现 —— 三层架构清晰，编译流水线设计精良
2. **安全防护纵深** —— bcrypt + OIDC nonce + SSRF + 限流 + 审计，覆盖 OWASP Top 10
3. **部署友好** —— 单镜像 Docker 交付，非 root 运行，一键部署
4. **CI 质量门禁** —— 40 个 verify 脚本 + pip-audit + ruff + vue-tsc，门禁齐全
5. **企业级 SSO** —— OIDC + SAML + LDAP 三合一，覆盖主流企业认证场景

主要短板集中在**测试覆盖率（58%）**和**前端体验细节**（面包屑、全局搜索、i18n），建议 v0.2.0 优先补齐测试覆盖到 75%+，补全 Pipeline 相关测试，引入浏览器端到端测试。
