## 三、实测对比（沙箱环境）

### 3.1 MarkItDown 实测结果

测试环境：3 核 CPU / 5.8GB RAM / 无 GPU，Python 3.14

| 测试文件 | 结果 | 耗时 |
|---|---|---|
| sample.docx（5 段 + 代码块） | 标题、段落、代码块完整保留 | <0.1s |
| sample.xlsx（2 工作表） | 表格 → Markdown 表格，sheet 名 → 标题 | <0.1s |
| sample.html（9 元素） | 标题、段落、代码块、列表完整 | <0.1s |
| sample.md（10 元素） | 透传，结构完整 | <0.1s |

**结论**：MarkItDown 在 Office/HTML 格式上速度快、结构保留好，且无需 GPU。

### 3.2 Unstructured 实测结果

测试环境：同上，但启动时 NLTK 数据下载失败（沙箱外网受限），无法完成测试。

**结论**：Unstructured 依赖 NLTK 模型下载，离线部署需预置模型包。即使成功运行，据 Benchmark 数据，复杂表格/公式场景精度不够。

### 3.3 MinerU 安装限制

- 要求 Python 3.10–3.13（沙箱为 3.14，无法安装）
- 生产环境需 GPU（8GB+），CPU pipeline 模式可用但较慢（0.3 页/秒）
- 模型需额外下载（~2GB），支持离线部署

## 六、最终选型方案

基于以上对比，本项目采用**分层组合**策略：

```
输入文件
│
├── PDF / 图片 → MinerU（VLM+OCR，高精度）┐
│                                         ├── Markdown ──→ 自研 Markdown 解析器
├── .docx/.xlsx/.pptx/.html → MarkItDown ┘      │
│                                         提取结构化元素
├── .sql → sqlparse（自研，DDL/DML 语义）       │
│                                         ↓
├── .md → 自研 Markdown 解析器          ParsedDocument
│                                         ↓
└── .txt → 自研 Text 解析器         知识抽取引擎
```

### 设计原则

1. **MarkItDown 为主力**（MIT 许可，无 GPU 依赖）——覆盖 Word/Excel/PPT/HTML
2. **MinerU 为 PDF 专项**（VLM 高精度，含公式/表格/109 语言 OCR）——仅 PDF/图片
3. **自研补位**——SQL（sqlparse 语义）、Markdown（结构提取）、TXT（纯文本）
4. **Unstructured 为备选**——当 MarkItDown 或 MinerU 不可用时的 fallback
5. **统一输出**：所有解析器最终输出 ParsedDocument，下游无感知

### 对已有代码的影响

- 删除自研的 `word_parser.py`、`excel_parser.py`、`html_parser.py`（由 MarkItDown 替代）
- 保留并增强 `markdown_parser.py`（作为 MarkItDown/MinerU 输出的下游解析器）
- 保留 `sql_parser.py`（语义特殊，无替代）
- 保留 `text_parser.py`（零依赖最快）
- 新增 `markitdown_adapter.py`（统一适配器）
- 新增 `mineru_adapter.py`（PDF 专项，Python 3.10-3.13 环境）
- 新增 `unstructured_adapter.py`（备选 fallback）
