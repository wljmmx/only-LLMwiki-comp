# 第一部分　V2.0 方案评审

## 1. 评审方法与范围

评审以"可落地执行"为标准，逐章核对 V2.0 文档的：①内部一致性；②技术选型的明确性与可行性；③需求可验收性；④关键创新点（知识编译、多 Agent、自愈合）的设计完备度；⑤部署与路线图的可执行性；⑥风险识别的完整性；⑦缺失的工程化要素（测试、CI/CD、安全、审计、回滚、评估）。

评审结论：**方向正确、视野完整，但作为执行基线尚不可直接落地**。主要问题集中在"选型未收敛、关键模块缺乏算法级设计、非功能指标不可度量、工程化要素缺失、路线图与资源不匹配"五个方面。

## 3. 逐项评审

### 3.1 架构与选型（对应 V2.0 第四、五章）

**优点**：六层分层清晰，组件均为成熟开源项目，本地化思路正确。

**问题**：
- **选型未收敛**：每个"或"都意味着两套部署、两套运维、两套技能要求，与"团队技能不足"风险直接冲突。必须做主备决策。
- **Wiki.js 定位过载**：V2.0 把 Wiki.js 同时作为"内容库 + 权限源 + 双向同步端"。Wiki.js 的强项是知识树与版本管理，但其 API 不适合做高频程序化双写。建议降级为"人工/半自动知识浏览端"，程序化内容主库改为自研（基于 Markdown + Git 或 PG）。
- **状态服务 HA 缺位**：Neo4j Community 单节点、Wiki.js 单节点、Milvus 单节点均不满足 NF2"冗余设计"。需明确 HA 方案或下调 NF2 措辞。
- **GPU 推理未量化**：见 D2，必须给出显存与并发对应表。

### 3.2 功能需求 F1–F7（对应 V2.0 第二章）

**问题**：每条功能只有描述，无**验收标准**与**度量项**。例如 F4"自然语言文档生成"未定义：单文档生成时长上限、章节结构完整率、引用回填率、用户修改率。
**修订要求**：每条 F 增补"验收标准 + 度量指标 + 优先级"，见第二部分第 5 节。

### 3.3 非功能需求 NF1–NF6（对应 V2.0 第三章）

- **NF1**：须明确"2s"是 P99 还是均值、是否含 LLM 调用、在何种数据规模下。建议拆为：向量检索 P99<500ms（100 万向量）、图查询 P99<800ms（100 万节点/边）、端到端 RAG（含重排）P99<2s、文档生成异步且单任务 P95<15min。
- **NF2**："冗余设计"与单节点状态服务矛盾，需逐组件列 HA 等级。
- **NF3**："权限继承自 Wiki.js"不可行（见 D4），需定义统一身份层。
- **NF5**：扩展点（解析器、本体、Agent）需定义插件接口契约，否则不可扩展。

### 3.4 知识编译引擎（对应 V2.0 第 6.2 节，核心创新）

**问题**：仅有流程描述，缺算法与阈值。需补：
1. **触发策略**：事件触发（新实体入图置信度≥θ1）、批量触发（每 N 条新知识或 T 时间窗）、手动触发。三者优先级与去重。
2. **去重与合并算法**：基于嵌入相似度（cos≥θ2）+ 结构化属性匹配 + LLM 语义判定三级；冲突时按"权威源 > 更新时间 > 多数投票"仲裁，并产出冲突报告。
3. **置信度门控**：抽取置信度≥θ3 自动入图；θ4≤c<θ3 进"建议审查"；c<θ4 丢弃并记录。θ 默认值与可调范围需给出。
4. **源真值模型**：图谱为结构化知识源真值；Wiki 为编译产物（可重建）；原文档为证据源真值。三者关系单向收敛，禁止 Wiki 反向写图（V2.0 的"双向链接"应理解为"引用"而非"反写"）。
5. **重编译影响分析**：图节点变更 → BFS 遍历关联 Wiki 页面 → 入重编译队列 → 乐观锁更新版本。

### 3.5 多 Agent 流水线（对应 V2.0 第 6.4 节）

**问题**：
- 角色重叠：审核（技术准确性）与校对（格式）边界模糊，易互相推诿。
- 退出条件缺失：最多三次修正后仍不通过的处置未定义。
- 成本无界：无单任务 Token 上限、无超时、无降级策略。

**修订**：见第二部分第 8 节，明确 5 角色（含意图理解）、状态机、退出条件、Token 预算、人工介入协议。

### 3.6 自愈合模块（对应 V2.0 第 6.5 节）

**问题**：
- "自愈合"与"自演化"未区分边界。建议：自愈合=恢复已存在正确状态（解析失败重试、冲突仲裁）；自演化=产生新知识结构（编译新 Wiki、提炼新经验模板）。两者 SLA 与人工介入阈值不同。
- 冲突仲裁"按权威性和更新时间"中"权威性"未定义度量。
- 一致性监控"定时对比图与 Wiki"缺乏具体校验项。

**修订**：见第二部分第 9 节，给出权威性评分公式与一致性检查清单。

### 3.7 部署与硬件（对应 V2.0 第七章）

**问题**：
- GPU 显存未量化（D2）。
- Wiki.js 依赖 PostgreSQL 未列出。
- 网络拓扑、端口、数据卷、备份窗口未定义。
- "无状态服务多副本"与 GPU 资源稀缺矛盾（LLM 推理无法简单多副本）。

**修订**：见第二部分第 12 节。

### 3.8 路线图（对应 V2.0 第八章）

**问题**：未绑定团队规模；阶段一 8 周目标过载；无测试/验收周；无灰度/试点安排。

**修订**：见第二部分第 13 节，按 6 人核心团队重排，每阶段增验收周与试点。

### 3.9 风险评估（对应 V2.0 第九章）

**缺失风险**：知识图谱污染与漂移、模型版本漂移导致抽取不一致、状态服务单点故障、AI 产出合规与版权、Token 成本失控、双写冲突导致数据丢失。见第二部分第 14 节补全。

### 3.10 缺失的工程化要素

### 6.2 数据流与源真值模型

```
原文档（证据源真值）
   │ 解析+抽取
   ▼
知识图谱 Neo4j（结构化知识源真值）── 嵌入 ──▶ Milvus（检索）
   │ 知识编译引擎（事件/批量/手动触发）
   ▼
LLM-Wiki 内容库 PG+Git（编译产物，可重建）──▶ Wiki.js（只读浏览）
   │
   ▼
文档生成流水线（Graph RAG 检索 ─▶ 多 Agent ─▶ Pandoc 渲染）
```

**源真值规则**（解决 D3）：
- 原文档 → 图谱：单向，图谱不反写原文档。
- 图谱 → Wiki：单向编译，Wiki 不反写图谱（V2.0"双向链接"重新定义为"Wiki 引用图节点 ID"，编辑 Wiki 仅更新 Wiki 自身，不触发图谱变更）。
- 图谱变更 → 触发受影响 Wiki 重编译（单向）。
- 任何反向修正走"人工建议"队列，由知识工程师审核后写图。

### 6.3 统一身份与权限（解决 D4）

- **身份源**：独立部署的 Keycloak（或复用企业 LDAP/OIDC）。
- **权限模型**：RBAC + 资源级 ACL，统一在网关层（Kong/APISIX）鉴权，下游服务信任网关透传的 JWT。
- **Wiki.js 不再作为权限源**，而是从 Keycloak 同步用户/组（SCIM 或定时同步）。
- **权限矩阵**（示例）：

| 角色 | 浏览图谱 | 编辑图谱 | 审核 Wiki | 触发生成 | 管理模型 |
|---|---|---|---|---|---|
| 普通用户 | ✓（本域） | ✗ | ✗ | ✓ | ✗ |
| 运维专家 | ✓ | 建议 | ✓ | ✓ | ✗ |
| 知识工程师 | ✓ | ✓ | ✓ | ✓ | ✗ |
| 系统管理员 | ✓ | ✓ | ✓ | ✓ | ✓ |

## 7. 知识编译引擎详细设计（核心创新，解决 D5）

### 7.1 触发策略

| 触发类型 | 条件 | 去重 | 优先级 |
|---|---|---|---|
| 事件触发 | 单条新知识置信度 c≥θ_event（默认 0.85）且影响实体度≥k（默认 3） | 同实体 5min 窗口合并 | 高 |
| 批量触发 | 每 N=50 条新知识 或 T=2h，取先到 | 同主题聚类合并 | 中 |
| 手动触发 | 知识工程师在 UI 触发 | 不去重 | 高 |
| 变更触发 | 图节点属性变更（版本号变化） | BFS 影响集合并批 | 高 |

### 7.2 去重与合并算法（三级）

1. **嵌入级**：新知识 e 与候选集 C 计算余弦相似，cos≥θ_dedup（默认 0.92）进入下一级。
2. **结构级**：实体类型、关键属性、关系三元组结构化匹配（Jaccard≥0.8）。
3. **语义级**：LLM 判定"是否同一知识"，输出 merge/update/skip + 理由。

### 7.3 冲突仲裁

冲突定义：同一实体同一属性出现≥2 个不一致值。
权威性评分：`Authority = 0.5*SourceWeight + 0.3*Recency + 0.2*Consensus`
- SourceWeight：人工标注文档=1.0，系统手册=0.9，历史工单=0.7，AI 推断=0.4。
- Recency：归一化更新时间新近度。
- Consensus：支持该值的来源数占比。

仲裁：取 Authority 最高值；差值<0.15 时不自动仲裁，进人工队列并产出冲突报告（含三方证据）。

### 7.4 置信度门控

| 置信度区间 | 处置 |
|---|---|
| c≥0.85 | 自动入图，触发编译 |
| 0.60≤c<0.85 | 入"建议审查"队列，UI 推送 |
| c<0.60 | 丢弃，记录日志供复盘 |

阈值可通过配置热更新，并纳入 A/B 评测。

### 7.5 重编译影响分析

图节点变更 → 取关联 Wiki 页面（通过引用关系表）→ 计算变更影响度（属性变更=轻度、关系变更=重度）→ 入重编译队列（乐观锁版本号）→ 编译完成 diff 审核 → 发布或回滚。

## 8. 多 Agent 流水线详细设计（解决 D6）

### 8.1 角色与职责（5 Agent，去重叠）

| Agent | 职责 | 输入 | 输出 | 不负责 |
|---|---|---|---|---|
| 意图理解 | 解析用户需求→结构化需求规格 | 自然语言 | {主题, 章节, 深度, 受众, 引用要求} | 检索 |
| 大纲生成 | Graph RAG 检索 + 大纲合成 | 需求规格 | 大纲 JSON（含每节检索证据 ID + 置信度） | 正文 |
| 生成 | 按大纲逐节生成正文 | 大纲+证据 | Markdown 正文 | 事实核对 |
| 审核 | **事实与技术准确性**核对（对照证据） | 正文+证据 | 结构化修改意见（定位+问题+建议） | 格式 |
| 校对 | **格式与一致性**（标题层级、术语、引用编号、图表） | 正文 | 格式修改意见 | 事实 |
| 修改 | 执行审核+校对意见 | 正文+意见 | 修订正文 | 重写大纲 |

注：审核与校对**串行**（先事实后格式），避免职责重叠。

### 8.2 状态机

```
意图理解 ──▶ 大纲生成 ──▶ 用户确认 ──▶ 生成 ──▶ 审核 ──▶ 校对 ──▶ 修改
                                   │                              │
                                   │全自动                         │ ≤3 次
                                   ▼                              ▼
                              (跳过确认)                    通过?──▶ 渲染输出
                                                                 │否
                                                                 ▼
                                                            人工介入队列
```

### 8.3 退出条件与预算

- **自迭代上限**：3 次。每次审核输出"通过/不通过+意见"；累计 3 次不通过转人工。
- **Token 预算**：单任务软上限 120k、硬上限 200k；超软上限降级（关闭重排、缩减证据数）；超硬上限中止并告警。
- **超时**：单 Agent 5min，全流水线 30min。
- **人工介入协议**：转人工时冻结任务、保留全量中间态、生成介入摘要（含卡点 Agent、未解决意见）。

## 9. 自愈合与自演化模块详细设计（解决 D11）

### 9.1 边界划分

| 类别 | 定义 | 人工介入阈值 |
|---|---|---|
| 自愈合 | 恢复已存在的正确状态（解析失败重试、冲突仲裁、一致性修复） | 仲裁驳回率>20% 时人工复核 |
| 自演化 | 产生新知识结构（编译新 Wiki、提炼经验模板、本体扩展建议） | 本体扩展必须人工确认 |

### 9.2 自愈合清单

| 场景 | 检测 | 自愈动作 | 失败兜底 |
|---|---|---|---|
| 解析失败 | 解析器异常 | 切换备选解析器（Unstructured→MinerU） | 入失败队列+告警 |
| 抽取低置信 | c<0.60 占比>30% | 触发 Few-shot 重试 + 切换 prompt | 转人工抽检 |
| 知识冲突 | 同实体多值 | §7.3 仲裁 | 差值小则人工 |
| 图-Wiki 不一致 | 定时校验 | 触发重编译 | 标记待修复 |
| 分类错误 | 用户反馈"不相关" | 重新分类+影响传播 | 人工确认 |

### 9.3 一致性检查清单（每周）

- 图节点属性 vs Wiki 引用值是否一致。
- Wiki 引用的图节点 ID 是否仍存在（悬空引用）。
- 向量索引与图谱实体数量偏差是否>5%。
- 编译产物的证据链是否可回溯到原文档。

### 9.4 "Karpathy 记忆分类"工程映射（澄清 D11）

借鉴其"参数记忆 vs 上下文记忆"的二分：
- **参数记忆**：不写入模型权重，而是固化在图谱与 Wiki（结构化长期记忆）。
- **上下文记忆**：Agent 工作流上下文（短期，任务结束清理，关键决策落审计表）。
- **情景记忆**：图节点属性 + 结构化案例模板（故障案例 JSON Schema）。
V2.0 的四分类保留，但明确"工作记忆"非纯临时——关键决策必须持久化到审计表以支持复盘与恢复。

## 10. 数据安全、权限与审计（新增，解决 D8/D10）

- **PII/敏感数据**：入库前经本地敏感词/正则脱敏服务；脱敏规则可配置；脱敏前后映射存加密表（仅审计可查）。
- **审计日志**：所有 AI 决策（抽取入图、编译发布、仲裁、生成任务）写入审计表，含 actor、动作、前/后值、证据 ID、模型版本、prompt hash，保留≥1 年。
- **网络隔离**：除前端 443 外全部内网；出站默认拒绝；模型与向量库仅允许应用网段访问。
- **密钥管理**：Vault 或本地 secrets；禁止明文配置。
- **数据导出**：受限角色可导出，导出动作入审计。

## 11. 测试、评估与质量保障（新增，解决 D8）

### 11.1 测试分层

| 层级 | 范围 | 工具 | 准入 |
|---|---|---|---|
| 单测 | 解析器、抽取 JSON Schema、仲裁算法 | pytest | 覆盖率≥80% |
| 集测 | 解析→抽取→入图→编译全链路 | pytest + testcontainers | 关键路径全绿 |
| 评测集 | 运维文档黄金集（≥200 标注样本） | 自研评测脚本 | 抽取 F1≥0.85 |
| 压测 | RAG/生成并发 | Locust | NF1 达标 |
| 人工抽检 | Wiki/文档产出 | 抽检表 | 抽检率 5% |

### 11.2 知识质量评估指标（新增）

- 抽取 F1、入图准确率、冲突率、悬空引用率、Wiki 编译采纳率、文档用户修改率、引用回填率。
- 仪表盘按周出报告，异常指标触发告警。

### 11.3 CI/CD 与回滚

- CI：lint + 单测 + 集测 + 镜像构建；评测集作为合并门控。
- CD：GitOps（ArgoCD/Flux），灰度发布（10%→50%→100%）。
- 回滚：图谱与 Wiki 每次发布保留版本快照（PG 时间表 + Git tag）；支持按版本回滚；模型回滚通过镜像 tag 切换。

## 12. 部署与硬件（修订版，解决 D2/D12）

### 12.1 GPU 显存与并发

| 模型 | 精度 | 显存 | 推荐卡 | 并发（vLLM） |
|---|---|---|---|---|
| Qwen2.5-14B-Instruct | FP16 | ~28GB | A100 40G/80G 或 2×4090 张量并行 | 8–16 流 |
| Qwen2.5-14B-Instruct | INT8 | ~16GB | A10 24G / 4090 24G | 4–8 流 |
| Qwen2.5-7B（降级） | FP16 | ~16GB | A10 24G / 4090 24G | 8–16 流 |
| BGE-M3 | FP16 | ~2.4GB | 与 LLM 共卡或 CPU 节点 | 高 |

**决策**：生产主选 A100 40G ×1 跑 14B FP16；预算受限用 2×4090 张量并行或 14B INT8 单卡。**A10/4090 24G 跑 14B FP16 不可行**，必须降精度或换卡。

### 12.2 容器与网络拓扑

- 网络：单 docker network / K8s namespace；仅前端 Service 暴露 443。
- 端口：内部全部 8080/9090 等约定端口，外部不可达。
- 数据卷：Neo4j、Milvus、PG、Ollama 模型目录独立 PVC，SSD。
- 备份：PG pg_basebackup 每 15min；Neo4j dump 每 1h；Milvus 快照每 6h；保留 7 天。

### 12.3 HA 等级

| 服务 | HA 方式 | RPO | RTO |
|---|---|---|---|
| PG | 主从 + Patroni | 15min | 5min |
| Milvus | 集群多副本 | 6h | 30min |
| Neo4j | 单节点 + 定时 dump（阶段一） | 1h | 30min |
| 自研服务 | 多副本 | — | — |
| LLM 推理 | 单 GPU 节点（NF1d 内） | — | 切备模型 |

> 阶段一 Neo4j 单节点不满足 NF2"冗余"，明确为**已知降级项**，阶段三评估升级 Enterprise 或迁 NebulaGraph。
